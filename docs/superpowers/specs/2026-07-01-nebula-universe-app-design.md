## 1. 设计目标

### 1.1 核心目标

打造一个单文件、可静态部署、具有商业级视觉质感的「星云编年史」主站。新页面需要：

1. 把 `index.html` 的沉浸式三阶段流程与 `nebula-chronicle.html` 的封面/主界面融合到一个新文件 `app.html` 中。
2. 保留 `nebula-chronicle.html` 封面的全部动态效果，并在此基础上按需加入音频可视化、全息卡片、鼠标视差等创新元素。
3. 加载阶段游戏化，像游戏启动画面一样有动态人物/能量场/进度叙事。
4. 主功能面从「螺旋时间轴」升级为「3D 动漫星系」：每部动漫作品是一颗恒星，每个 AniList 官方流派是一片星云，所有星云围绕具象化的「编年史核心」旋转。
5. 通过 AniList API 预拉取 500–1000 部经典+热门作品，生成静态 JSON 数据文件，前端直接加载，不依赖实时 API。
6. 作品详情保留「看动漫」按钮，跳转 `watch.html`，本次不深入改造观影页。
7. 视觉风格：二次元科幻宇宙 + 唯美星空治愈 + 手绘少女感。

### 1.2 项目背景与定位

- **项目背景**：「星云编年史」是一个面向动漫爱好者的沉浸式网页体验项目，以 1963 年《铁臂阿童木》开启的日本动画纪元为宇宙起点，将动画史转化为可探索的虚拟星系。
- **当前状态**：项目仅有静态前端页面（`index.html`、`nebula-chronicle.html`、`watch.html`），无后端服务。数据通过构建时 Node.js 脚本从 AniList GraphQL API 拉取并生成静态 JSON。
- **平台假设**：基于 TRAE Work 平台与 Kimi K2.7 Code 模型进行协同开发，利用其 Plan/Spec 白盒机制与多轮 Agentic 能力控制代码质量、降低误改风险。前端采用 Vue 3 + Vite 构建，兼顾组件化、可维护性与静态部署能力。
- **产品定位**：短期内作为开放展示/宣传项目，长期可探索与动画制作方合作、周边电商、企业定制等商业化路径。

### 1.3 目标用户与场景

| 用户群体 | 典型场景 | 体验重点 |
|---------|---------|---------|
| 国内外动漫爱好者 | 桌面端沉浸式浏览动画史 | 视觉冲击、流畅交互、作品发现 |
| 技术社区用户 / 前端开发者 | 查看源码、学习 Three.js/WebGL 实现 | 代码质量、创新性、可复用性 |
| 展会/赛事参观者 | 现场大屏演示、触摸交互 | 快速吸引注意力、稳定运行 |
| 移动端用户 | 手机/平板随时浏览 | 简化但完整的星图/列表体验 |

### 1.4 成功标准

- **视觉**：页面呈现「远超参考项目诗云」的宏大宇宙感，商业级视觉质感。
- **性能**：桌面端动画流畅度 ≥ 60 FPS，首屏加载 ≤ 3 秒；移动端可降级浏览不崩溃。
- **交互**：鼠标拖拽、缩放、hover、点击、搜索/筛选、观影跳转形成完整闭环。
- **数据**：覆盖 500–1000 部精选作品，按 AniList 官方流派完整分类。
- **传播**：开源后形成技术圈与二次元社区双重话题，具备二次传播价值。

## 4. 宇宙世界观与视觉语言

### 4.1 次元核心（Chronicle Core）

- **定位**：星系中心，象征 1963 年《铁臂阿童木》开启的日本动画纪元。
- **形态**：二次元科幻宇宙 + 唯美治愈的结合体——一颗缓慢旋转的「星核」。
  - 外层：柔和的多层光晕（手绘感边缘，类似水彩光斑）。
  - 中层：缓慢流动的能量环，带霓虹描边。
  - 内核：半透明的发光球体，内部有隐约漂浮的动画符号（胶片、音符、羽毛笔等极简手绘元素）。
- **动效**：自转 + 呼吸脉动 + 能量粒子缓慢向外飘散。

### 4.2 流派星云

- 每个 AniList 官方流派对应一片星云。
- 星云形态：使用 Three.js 粒子或着色器实现的体积雾，颜色由流派主题色决定。
- 布局：星云围绕核心分布在不同轨道平面上，形成类似银河悬臂的结构。
- 命名示例：科幻星云、机战星云、治愈星云、悬疑星云、热血星云等。

### 4.3 作品恒星

- 每部动漫作品渲染为一颗星点。
- 视觉编码：
  - **颜色**：流派主题色。
  - **大小**：综合评分 + 历史影响力。
  - **亮度**：人气/流行度。
  - **位置**：所属星云内部，按年代/影响力分层。
- 交互：hover 放大 + 显示工具提示；click 打开详情面板。

### 4.4 星座连线

- 同系列、同导演、同工作室或共享高频标签的作品之间用微弱光带连接。
- hover 某颗星时，高亮其相关连线与邻星。

### 4.5 视觉风格关键词

- **二次元科幻宇宙**：霓虹青、紫、粉、蓝；全息、扫描线、数据流。
- **唯美星空治愈**：柔和光晕、梦幻渐变、羽毛/花瓣/星光等手绘粒子。
- **手绘少女感**：圆角、手写体小字、柔和描边、淡彩点缀、少量蕾丝/蝴蝶结等装饰元素（点到为止，不喧宾夺主）。

## 5. 四阶段详细设计

### 5.1 Phase 0：次元启动（Loading）

- **视觉**：
  - 背景：深空星云 + 微弱粒子。
  - 中央：一个手绘风格的角色剪影/能量体（可用 SVG + CSS 动画实现），被旋转的能量环包围。
  - 下方：环形进度条 + 百分比数字 + 随机切换的 tips（如“正在连接星云数据库…”、“正在点亮恒星…”、“正在绘制星云图谱…”）。
- **行为**：
  - 预加载 `data/anime-corpus.json`、`data/genre-manifest.json`、关键字体、封面视频首帧。
  - 加载完成后自动进入 Phase 1。
- **音频**：启动低沉的环境音（可选，若用户开启音乐）。

### 5.2 Phase 1：全息展示（Showcase）

- **视觉**：
  - 中央：缩小的次元核心。
  - 周围：6–10 张代表性作品卡片以全息投影形式环绕漂浮。
  - 卡片效果：半透明玻璃质感、扫描线、顶部高光、微弱 RGB 分离（glitch 但不强烈）、缓慢上下浮动。
- **行为**：
  - 卡片自动缓慢旋转展示。
  - 5–8 秒后自动进入 Phase 2；也可点击跳过。

### 5.3 Phase 2：星云入口（Entrance）

- **视觉**：完整移植 `nebula-chronicle.html` 封面：
  - 星云背景 Canvas
  - 双视频交叉淡入淡出轮播
  - 动态光斑（orb）
  - 星点画布
  - 流星效果
  - 漂浮粒子
  - 次元传送门光环
  - 标题与「开始旅程」按钮
- **新增元素（按需）**：
  - 音频可视化：背景光斑/传送门随环境音节奏脉动。
  - 鼠标视差：背景层随鼠标轻微移动，增强空间感。
  - 全息标题：标题文字带轻微扫描线和 RGB 分离。
- **行为**：
  - 点击「开始旅程」后，全屏白闪/星爆过渡，进入 Phase 3。

### 5.4 Phase 3：动漫星系（Galaxy）

- **视觉**：
  - Three.js 全屏 3D 场景。
  - 次元核心在中心。
  - 流派星云分布在不同轨道。
  - 作品恒星在各自星云内缓慢公转。
  - 远处有淡淡的星尘背景。
- **交互**：
  - 鼠标拖拽：旋转视角。
  - 滚轮：缩放。
  - hover 星点：显示工具提示（标题、年份、流派、评分）。
  - click 星点：打开右侧/中央详情面板，展示封面、简介、标签、评分、工作室，以及「看动漫」按钮。
- **HUD**：
  - 顶部：Logo、搜索框、流派筛选 chips。
  - 右上角：音乐开关、语音开关、帮助/说明。
  - 左下角：当前视角信息 / 作品计数。

## 6. 数据策略

### 6.1 数据来源

- 使用 **AniList GraphQL API**（公开、免费、无需授权）。
- 拉取维度：
  - 按评分/人气排序的 TOP 作品。
  - 按年代补充经典作品（1960s–2020s）。
  - 保证每个 AniList 官方流派都有代表作品。
- 目标数量：500–1000 部作品。

### 6.2 构建时脚本

- 文件：`scripts/fetch-anilist.js`（Node.js）。
- 职责：
  1. 调用 AniList API 分页拉取作品。
  2. 清洗、去重、分类。
  3. 生成 `data/anime-corpus.json`。
  4. 生成 `data/genre-manifest.json`（流派名称、主题色、星云位置参数）。
- 运行时机：
  - 开发阶段手动运行更新数据。
  - CI/CD 中可自动运行。
  - 运行时前端不直接调用 AniList，保证静态部署可用。

### 6.3 数据字段

```json
{
  "id": 1,
  "titleRomaji": "Cowboy Bebop",
  "titleNative": "カウボーイビバップ",
  "titleChinese": "星际牛仔",
  "year": 1998,
  "genres": ["Action", "Adventure", "Drama", "Sci-Fi"],
  "tags": ["Space", "Bounty Hunters", "Jazz"],
  "description": "...",
  "coverImage": "https://s4.anilist.co/file/...",
  "averageScore": 86,
  "popularity": 450000,
  "studios": ["Sunrise"],
  "director": "Shinichiro Watanabe",
  "format": "TV",
  "episodes": 26
}
```

### 6.4 离线兜底

- 若 AniList 图片加载失败，使用本地 `images/generated/nebula-bg.jpg` 或生成色块占位。
- 保留现有 `rawAnimeData` 作为最小兜底数据集。

## 7. 技术栈

### 7.1 前端运行时

| 层级 | 技术 |
|------|------|
| 框架 | Vue 3（Composition API）+ Vite |
| 页面 | `index.html` + `src/App.vue` 及组件化页面 |
| 3D 渲染 | Three.js（npm 包或 CDN，按 Vite 配置决定） |
| 数据 | AniList GraphQL + 构建脚本生成 JSON |
| 字体 | Google Fonts（Orbitron、Noto Sans SC、ZCOOL XiaoWei） |
| 动画 | CSS Animation、Canvas 2D、WebGL Shader、GSAP（可选） |
| 音频 | Web Audio API 生成氛围音 + 可选 BGM |
| 静态部署 | GitHub Pages / Vercel / Cloudflare Pages / 任意静态托管 |

### 7.2 开发协同与 AI 能力

- **开发平台**：TRAE Work（Code 模式），利用 Plan/Spec 白盒执行机制，在编码前先形成设计与实现计划，降低 AI 误改风险。
- **主力模型**：Kimi K2.7 Code，擅长长上下文编程任务、批量生成复杂前端逻辑与 Three.js/WebGL 代码，支持多模态输入（可将概念图、参考视频作为输入生成对应代码）。
- **协作方式**：
  1. 使用 TRAE Work 的 Plan/Spec 模式拆分里程碑；
  2. 由 Kimi 生成模块代码并通过静态检查；
  3. 人工审查关键交互与性能路径；
  4. 逐步合并到 `app.html` 并本地预览验证。

### 7.3 生成式素材（可选）

- 星云背景、粒子纹理、角色剪影等视觉元素可结合 Stable Diffusion / TRAE 图像生成能力创作，并在文档中标注「AI 生成，非官方画作」。
- 背景音乐优先使用免版税音乐或 Web Audio API 程序化生成，避免版权风险。

## 10. 风险、约束与缓解措施

### 10.1 技术与运营风险

| 风险 | 影响 | 缓解方案 | 备选方案 |
|------|------|----------|----------|
| AniList API 速率限制 | 拉取 1000 条较慢 | 构建时分页 + 指数退避；数据生成后不再实时请求 | 降低数据量或分阶段加载；使用其他公开动漫数据库 |
| 外部图片加载失败 | 星系中作品封面无法显示 | 本地兜底占位图；图片懒加载；使用 AI 生成替代图 | 仅显示色块 + 文字信息 |
| 3D 性能不足 | 低端设备卡顿 | LOD、粒子数量分级、提供「轻量模式」开关 | 移动端强制 2D 星图/列表；关闭次要粒子效果 |
| 单文件过大 | 首屏加载慢 | 数据 JSON 异步加载；代码模块化；关键 CSS 内联 | 拆分为多页面或按需加载阶段 |
| 手势/摄像头兼容性 | MediaPipe 部分设备不可用 | 鼠标交互作为完整 fallback | 移除手势模块，仅保留鼠标/触摸 |

### 10.2 合规与内容风险

| 风险 | 影响 | 缓解方案 | 备选方案 |
|------|------|----------|----------|
| 版权合规（封面、BGM） | 潜在侵权纠纷 | 优先使用 AniList 官方图片并遵守 API 条款；BGM 使用免版税音乐或 Web Audio 程序化生成；AI 生成素材标注来源 | 移除争议素材，使用纯生成式替代 |
| AI 生成内容准确性 | 简介/数据错误误导用户 | AI 仅辅助生成，人工审核关键文案；优先使用官方资料 | 禁用 AI 生成，仅保留官方/手动内容 |
| 隐私与安全 | 用户行为数据合规 | 不收集敏感信息；若后续加入登录/收藏，补充隐私政策；定期审查第三方依赖 | 关闭需要登录的功能 |
| 运营合规 | 中国 ICP 等法规 | 静态托管前确认备案要求；传播时避免敏感/虚假宣传 | 仅在海外平台发布 |

## 12. 设计决策记录

- **不引入 Rust 后端**：用户确认目前只有静态网页，改用构建时 Node 脚本生成 JSON。
- **数据量 500–1000 部**：平衡视觉效果与性能，避免一次性拉取全量 AniList 数据。
- **分类采用 AniList 官方流派**：用户选择，后续可扩展为多元宇宙维度。
- **观影入口跳转 `watch.html`**：用户确认当前不改观影页。
- **视觉风格融合**：二次元科幻宇宙 + 唯美星空治愈 + 手绘少女感。
- **编年史核心具象化**：设计为带多层手绘光晕、能量环、内部漂浮动画符号的实体星核。
- **封面特效保留 + 增强**：完整移植 nebula-chronicle 封面动效，并叠加音频可视化、全息标题、鼠标视差。
- **AI 辅助开发**：使用 TRAE Work + Kimi K2.7 Code 进行 Plan/Spec 驱动的协同开发。

## 13. 创新点与差异化

### 13.1 跨界视觉风格

融合「二次元 + 科幻 + 唯美治愈 + 手绘少女」四种风格，形成独一无二的视觉语言：
- **二次元科幻宇宙**：霓虹青、紫、粉、蓝，全息扫描线、数据流、能量环。
- **唯美星空治愈**：柔和光晕、梦幻渐变、羽毛/花瓣/星光等手绘粒子。
- **手绘少女感**：圆角 UI、手写体小字、柔和描边、淡彩点缀，点到为止不喧宾夺主。

### 13.2 游戏化加载与叙事

将传统 Loading 页设计成游戏启动界面：
- 中央角色剪影/能量体被旋转能量环包围。
- 环形进度条 + 百分比 + 随机切换的叙事 tips（如「正在连接星云数据库…」、「正在点亮恒星…」）。
- 低沉环境音随进度逐渐清晰，提升期待感。

### 13.3 3D 动漫星系体验

- 每部作品是一颗恒星，每个 AniList 流派是一片星云。
- 所有星云围绕具象化的「编年史核心」旋转，形成远超参考项目「诗云」的宏大宇宙感。
- 支持鼠标/触摸/手势多模态交互，hover 高亮星座连线，click 打开详情面板。

### 13.4 AI 辅助开发流程

- 利用 TRAE Work Plan/Spec 白盒机制，确保代码可追溯、可审查。
- 利用 Kimi K2.7 Code 的长上下文与多模态能力，快速生成 Three.js、WebGL、动画逻辑代码。
- 将 AI 协同开发本身作为项目亮点，在技术社区传播。

## 16. 可执行优化建议

### 16.1 AI 与开发流程

- 使用 Kimi K2.7 Code Plan/Spec 模式先生成复杂模块计划，再分步编码。

### 16.4 UI/UX 与增长
